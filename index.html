总结基于Bi—LSTM和CRF的医疗实体抽取

在整个实体抽取的过程中,大致可分为数据输入, 构建LSTM模型, 解码输出三部分

数据输入

一. 从txt文本中读取固定格式的数据，根据其格式解析即可获得原始的文本输入.即得到由文本组成的输入序列和“DIS”, “BOD”等标签组成的标记序列。

数据格式为：
输入序列: [ [主因上腹部、腰部疼痛1天入院。] ….]
标记序列: [[O O B-BOD M-BOD E-BOD O B-BOD E-BOD B-BEH E-BEH O O O O O]…. ]

二. 通过使用pandas库内建的DataFrame数据类型, 统计文本的词表和标签集合，并将原始的文本,标签转化为词表中文本对应的下标，和标签集合中对应的下标。得到输入文本序列和输出标签序列的数字表示。然后将全部病例按照0.8：0.2的比例划分为训练集和验证集

数据格式为:
输入序列: [[2, 3, 5, 6, 8, 1, 10, 17, 23, 29, 15, 20, 45, 58, 0 ]….]
标注序列： [[0, 0, 1, 2, 3, 0, 1, 3, 4, 6, 0, 0, 0, 0, 0] ……]

三. 将不等长的每个输入/输出序列(即正常的每一句话及其对应的标签序列)通过补0的方式调整其长度是8的倍数, 然后再切分成不定数量的长度为24,滑动长度为8的子序列，将子序列作为LSTM的基本输入

数据格式为：
输入序列：[[2, 3, 5, 6, 8, 1, 10, 17, 23, 29, 15, 20, 45, 58, 0, 0…] (补充0至长度为8的倍数)….]
标注序列： [[0, 0, 1, 2, 3, 0, 1, 3, 4, 6, 0, 0, 0, 0, 0, 0…] ……]

改进及问题

一. 在一开始使用单向的LSTM时,出现了如下的情况:

> 复方鲜竹沥液 标注为B-TRE M-TRE M-TRE M-TRE E-TRE E-TRE
    正确标注应为B-TRE M-TRE M-TRE M-TRE M-TRE E-TRE

将单向的LSTM改为双向之后, 边界的情况有所改善。说明双向的LSTM模型对标注的边界处理更好
二. 将标注序列的字典由原来按频率统计得到变为手动设置，使其相邻间有关联，即使其有相同后缀的标签相邻，训练集和测试集的准确率和召回率均有稳定的提升，可能是因为将标签规则化提升了模型的学习效率使得模型在同样的训练集中能够表现的更好

三. 使用CRF层解码前后的效果对比为:

事件抽取项目总结

发表于 2018-01-29

夹杂着各种心酸血泪史的回忆过程中,打算先按照时间顺序来说明项目的经过，然后再进行技术上的总结。

事件定义

   在开始bb之前，先来大致说一下事件抽取的背景好了。还不是因为老师拍脑袋想出来的策略，需要我来把甲方忽悠住

事件抽取任务的定义:

    事件抽取领域从上世纪80年代的开始发展,其主要的评测会议为:MUC和ACE。相比之下，MUC会议主要针对某一特定领域或者场景，基于预先定义好的模板进行识别。在MUC会议停办之后出现了ACE(Automatic Content Extraction)会议，并且 ACE更偏重于自动抽取语料中的实体，关系和事件等内容。
   根据ACE会议对事件定义：事件被分为8个类型和33个事件子类型(每种事件类别对应唯一的事件元素模板)。事件由事件触发词(Trigger)和描述事件结构的元素(Argument)构成.由触发词（Trigger）决定事件类别，进而确定事件模板中的元素进行抽取。如:”毛泽东1893年出生于湖南湘潭。”由“出生”为该事件的触发词，所触发的事件类型为Life,对应的子类别是Be—Born。原文中事件的3个组成元素为“毛泽东”，“1893年”和“湖南湘潭”，对应事件子类别模板的元素标签为：Person,Time和Place。

目前主流的解决方案:

ACE 会议评测任务定义了事件抽取两大核心子任务，即事件的检测与类型识别和事件论元角色的抽取。评测中事件抽取相关概念包括实体(entity)、实体描述(entitymention)、事件描述(eventmention)、事件论元角色 (eventargument)。其中，实体描述是文档中关于实体的描述，一个或多个实体描述构成一个实体；事件描述是文中具体描述事件的句子或片段；事件论元角色(也称事件元素)是在事件中担当某类角色的实体、数值或时间，包括事件参与者以及与事件相关的属性(如时间属性)。
由于论元语义信息难以获取,绝大多数句子级别的事件抽取系统还是沿用了语义角色标注(semantic role labeling,简称 SRL) 的方法,充分利用句法信息来识别事件实例及其论元。目前,触发词抽取和论元抽取的 F1 值一般分别在 50%~70%和 40%~60%左右。

    按照这种方式的定义骚气地将抽取式的表述任务将转化为了原文词语分类任务，进一步地将这个原文成分分类任务转化为常见的标注任务，通过类似于识别句中实体的操作识别句中事件的相关成分。
    上面的大段引用中提到过的句法分析，依存句法分析以及可用于序列标注的HMM，CRF，Bi-LSTM都可以用于事件抽取这个任务上。比如使用序列标注来识别医学事件 , 基于语义的中文事件触发词抽取联合模型 , A Language-Independent Neural Network for Event Detection , Joint Event Extraction via Structured Prediction with Global Feature , 以及逼乎的回答 (有过辣么一瞬间觉得逼乎拯救了我的性命)
    但是按照目前的共识即“有监督的要强于无监督，LSTM的效果要强于CRF”，我们采用了Bi-LSTM的方法。其实完全是因为老师知道我们刚做了一个Bi-LSTM+CRF的实体抽取,毕竟是决定能不能正确毕业的人要乖乖地好嘛
    在事件定义方面，怎么说呢，大家前期太划水以及什么都不会还找不到正确地解决方案，导致学长背锅被老师彻底怼翻车。然后发现按照项目进度走的话要凉凉了，于是中期动员整个组里的人手开始强行人力搞定，进而这种奇怪的主谓宾事件表述形式就被两位大佬拍脑袋定下来了。

这是在标注文档中的相关说明:

要求每个事件中的单独成分即主语、谓语或者是宾语，在原文中是一定是连续的如: 原文“不但要求每周观看不少于20集的动画 “。其对应事件的谓语写成“要求观看”，这个是不可以的，应该写成”观看”，这样在原文中连续的才可以。并且同一句话可能有多个事件，且这些事件的成分之间不可以有重叠
对于谓语,我们目前制定的标注规则是希望谓语尽可能地精简。如对于描述剧烈程度之类的如”增长迅猛”,”快速发展”之类的只会标注出”增长”和“发展”作为谓语，而程度副词是不标注的，同样修饰谓语的“了”，“起来”之类的同样不标注。但是对于否定词不，不能之类的是一定要标出来的。

嘛毕竟是拍脑袋想出来的，可行性和完备性其实并不是特别好，但是对吧毕竟大佬们想出来的……..瞎bb可能就需要我来提方案了，所以当然是选择听大佬的了。
